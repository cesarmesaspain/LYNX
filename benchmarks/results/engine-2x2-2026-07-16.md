## Verified findings

- Codebase Memory is materially faster at full indexing: 2.7× on the TypeScript
  LYNX repository and 7.3× on its C-heavy repository.
- LYNX uses 44% less peak memory on the C-heavy repository, while Codebase uses
  27% less on LYNX. Neither engine wins memory universally.
- LYNX preserved 1,127 C implementation nodes alongside same-name header
  declarations. Codebase preserved only 2 in the shared-schema probe. Before the
  LYNX fix, header and implementation QNs collided as well.
- LYNX emitted zero TypeScript→C name-only graph edges in both repositories.
  Codebase emitted 119 on LYNX and 60 on its own repository. Manual inspection
  confirms false matches including TypeScript `expect` → C `expect`, and generic
  identifiers such as `key`, `value`, `node`, `project`, and `data`.
- LYNX now resolves unique C header includes conservatively, keeps ambiguous
  basenames unresolved, and directs calls to a unique `.c` implementation with
  `imported-implementation` evidence. On the Codebase repository this produced
  16,029 such CALLS edges while retaining prototypes as declaration evidence.

## Decision

LYNX is currently stronger on the tested cross-language precision and C
declaration/definition identity invariants, but it is not yet superior overall:
the C extraction throughput gap remains too large. Raw edge volume is not a
success criterion.

A trial of the native Node `tree-sitter-c` binding was rejected after measurement:
it increased the full run from 28.99 s to 33.82 s (+16.7%). Parsing itself is not
the dominant cost; millions of per-node JS boundary operations are. The next
performance implementation should therefore be a fused native C extraction path
that emits compact definitions/calls/imports/usages in batches, followed by the
same precision-preserving resolver—not another parser wrapper.
